# StarClaw 多模型评测方案

> 基于 openEuler 日志异常检测 MCP 架构的深度评测

---

## 1️⃣ 背景

### 为什么做这个评测

随着大语言模型在软件开发领域的广泛应用，开发者面临一个核心问题：**如何选择最适合实际开发场景的模型？**

当前市面上的模型评测多集中于：
- 学术基准测试（如 MMLU、HumanEval）
- 通用能力评估
- 单一维度对比

**缺失的是**：针对开发者真实工作流的工程化效能评测，特别是：
- 日志分析与故障诊断能力
- 代码生成与执行效率
- 成本效益比分析
- 长期运行稳定性

本评测旨在填补这一空白，为开发者提供真正有决策价值的选型参考。

## 3️⃣ 评测场景/任务设计

### 任务一：日志异常检测

**输入**：1000行真实系统日志（含正常日志与异常日志混合）

**要求**：
1. 识别所有异常事件
2. 定位根本原因
3. 提供修复建议

### 任务二：代码生成与调试

**输入**：功能需求描述 + 错误日志

**要求**：
1. 生成符合需求的代码
2. 根据错误日志修复问题
3. 代码可直接运行

### 任务三：API 集成与错误处理

**输入**：API 文档 + 错误响应示例

**要求**：
1. 生成正确的 API 调用代码
2. 实现完善的错误处理
3. 处理边界情况

### 任务四：系统诊断与优化建议

**输入**：系统性能指标 + 用户反馈

**要求**：
1. 分析性能瓶颈
2. 提供优化建议
3. 预估优化效果

## 4️⃣ 评测方法

### 双场景对比测试

| 场景 | 输入 | 评测目标 |
|------|------|---------|
| **场景A：原始分析** | 完整原始日志（1000行） | 测试模型在无辅助下的"硬实力"和成本 |
| **场景B：预处理后分析** | 浓缩日志片段（50行高异常值） | 测试模型在高质量输入下的效能提升 |

### 评测步骤

1. **准备阶段**
   - 收集真实生产环境日志
   - 标注异常事件和根因
   - 构建测试数据集

2. **执行阶段**
   - 各模型分别处理原始日志
   - 各模型分别处理预处理后日志
   - 记录响应时间和 Token 消耗

3. **评估阶段**
   - 准确率评估：异常识别是否正确
   - 完整性评估：是否遗漏关键信息
   - 深度评估：根因分析是否深入
   - 可操作性评估：建议是否可执行

### 评分标准

| 维度 | 权重 | 评分标准 |
|------|------|---------|
| 问题定位准确率 | 30% | 正确识别异常事件比例 |
| 根因分析深度 | 25% | 分析是否触及根本原因 |
| 建议可操作性 | 20% | 建议是否可直接执行 |
| 响应时间 | 10% | 从请求到完整响应的时间 |
| Token 效率 | 15% | 单位 Token 产出的价值 |

### 核心效能指标

```
问题定位准确率 = 正确识别的异常数 / 总异常数 × 100%
根因分析深度 = 根因命中等级评分 (1-5分)
单次诊断成本 = API调用费用 (元)
价值成本比 = 综合质量评分 / 单次诊断成本
```

## 7️⃣ 开发者真实体验

### GLM-4-Flash 体验

**优点**：
- 中文理解准确，无需英文提示词
- 响应速度快，适合高频调用
- 价格低廉，成本可控

**不足**：
- 复杂推理偶尔出现逻辑跳跃
- 代码生成风格较为保守

**适用场景**：日常日志监控、快速问题排查

### DeepSeek-V3 体验

**优点**：
- 推理能力强，根因分析深入
- 开源可部署，数据安全有保障
- 长上下文支持完整日志分析

**不足**：
- 响应时间较长
- 本地部署需要较高配置

**适用场景**：复杂系统诊断、安全敏感场景

### Qwen2.5-Coder 体验

**优点**：
- 代码理解准确
- 生成的代码质量高
- 与阿里云生态集成好

**不足**：
- 非代码类问题分析能力一般
- 需要阿里云账号

**适用场景**：代码调试、API 集成

## 8️⃣ 评测整体总结

### 核心发现

1. **预处理的价值**：经过关键字过滤和聚类预处理后，所有模型的 Token 消耗降低约 90%，而分析质量反而提升（噪音减少）

2. **性价比之选**：GLM-4-Flash 在中文日志分析场景中性价比最高，适合作为 StarClaw 的主力模型

3. **能力边界**：单一模型难以覆盖所有场景，混合架构（预处理 + 多模型协作）是最佳实践

4. **工程化必要性**：纯靠大模型"蛮力"分析效率低下，必须引入预处理管道

### 选型建议

| 场景 | 推荐模型 | 理由 |
|------|---------|------|
| 日常监控告警 | GLM-4-Flash | 高性价比，快速响应 |
| 生产事故诊断 | DeepSeek-V3 | 推理深入，准确率高 |
| 代码问题排查 | Qwen2.5-Coder | 代码专精，修复建议准确 |
| 安全敏感场景 | DeepSeek-V3（本地部署） | 数据不出内网 |

### StarClaw 优化方向

基于评测结果，StarClaw 将采用以下架构：

```
用户请求 → 预处理管道 → 路由决策
                              ↓
              ┌───────────────┼───────────────┐
              ↓               ↓               ↓
         GLM-4-Flash    DeepSeek-V3    Qwen2.5-Coder
         (日常对话)      (复杂分析)       (代码问题)
              ↓               ↓               ↓
              └───────────────┼───────────────┘
                              ↓
                         结果聚合 → 用户响应
```
